git-stk

One commit, one PR. Ship faster with stacked changes.

Break down large features into reviewable chunks. Each commit becomes its own PR, stacked on the previous one.

curl -fsSL https://raw.githubusercontent.com/derdaele/git-stk/main/install.sh | bash
1

Create Your Stack

Create a feature branch and make commits. Each commit is a logical unit that will become a PR.

$ git switch -c my-feature-branch
$ git commit -m "feat: add user model"
$ git commit -m "feat: add auth endpoints"
$ git commit -m "feat: add login UI"
2

View Your Stack

See all commits and their PR status before exporting.

$ git stk view
┌─ main
│
├─●  abc1234  feat: add user model  [?→01]
│  <PR to be created>
│
├─●  def5678  feat: add auth endpoints  [?→02]
│  <PR to be created>
│
└─●  ghi9012  feat: add login UI  [?→03]
  <PR to be created>
3

Export to GitHub

Push branches and create PRs for each commit. Each PR depends on the one before it.

$ git stk export
Pushing branches...
✓ my-feature-branch--01
✓ my-feature-branch--02
✓ my-feature-branch--03
Creating PRs...
✓ PR #42: feat: add user model
✓ PR #43: feat: add auth endpoints
✓ PR #44: feat: add login UI
4

Update a Commit

Make changes to a commit in your stack and re-export to update the PR.

$ git commit --fixup def5678
$ git rebase -i --autosquash main
$ git stk export
Pushing branches...
✓ my-feature-branch--02 (force)
✓ my-feature-branch--03 (force)
Updating PRs...
✓ Updated PR #43
✓ Updated PR #44
5

Land a PR

Merge the bottom PR, wait for completion, pull changes, rebase your stack, and re-export.

$ git stk land

✓ Merge initiated
Waiting for merge to complete...
✓ PR merged successfully!
Pulling latest changes...
✓ Updated main
Rebasing stack...
✓ Rebased 2 commits on main
Re-exporting...
✓ Updated PR #43

Commands

git stk view
View the current stack of commits and their PR status. Shows a tree structure with commit SHAs, messages, slots, and PR links.
Safe to run: This command is read-only and never modifies your repository or remote state.
git stk export
Export the stack to GitHub by creating/updating branches and PRs for each commit in your stack.
Options
--draft
Create PRs as drafts
--ready
Mark PRs as ready for review
--open
Open created/updated PRs in browser
--dry-run
Show what would be done without making changes
git stk land
Merge the bottom PR in the stack, wait for merge completion, pull latest changes, rebase remaining commits, and re-export the stack.
Options
--skip-wait
Don't wait for merge to complete
git stk landed
Run post-merge operations after a PR has been manually merged outside of git-stk. Pulls changes, rebases, and re-exports.
git stk set slot <commit> <slot>
Manually assign a custom slot to a commit. The commit can be a SHA, stack index (1, 2, 3), or git ref (HEAD, branch name).
Warning: If the commit already has a slot with an exported PR, changing the slot will close the existing PR and create a new one on the next export. This is because the head ref (branch name) in a GitHub PR is set at creation time and cannot be changed.
Options
-y, --yes
Skip confirmation prompts

How It Works

What are Slots?

+

Slots are unique identifiers for commits that determine their branch names on GitHub. Each commit in your stack gets a slot, which is combined with your branch name to create the remote branch.

Auto-assigned slots: By default, git stk automatically assigns sequential numeric slots (01, 02, 03...) to your commits when you export.

Custom slots: You can assign custom string slots for better organization and clarity:

  • Link to issue trackers: PROJ-123, LINEAR-456
  • Descriptive names: add-auth, fix-validation
  • Any alphanumeric string with hyphens and underscores

The branch name format is: {your-branch}--{slot}

# Assign a custom slot
$ git stk set slot abc1234 PROJ-123
$ git stk export
✓ my-feature--PROJ-123

Where are the commit metadata stored?

+

All commit metadata (PR numbers, slots, etc.) are stored in git notes attached to each commit. Git notes are a built-in Git feature that allows you to attach arbitrary data to commits without modifying the commits themselves.

Key benefits:

  • Survive rebases: Git notes are automatically carried along when commits are rebased, ensuring your PR links stay intact
  • Persist locally: Metadata is stored in your local repository, not in commit messages
  • History-safe: Amending commits or interactive rebasing preserves the notes

You can view the raw notes using Git's built-in commands:

$ git notes --ref=refs/notes/git-stk show <commit>

What happens if I squash commits together?

+

When you squash multiple commits together (e.g., using git rebase -i), git stk handles the metadata as follows:

Metadata preservation: The metadata (slot and PR number) from the earliest commit in the squash is preserved on the resulting squashed commit. This ensures continuity for the primary PR.

Existing PRs: If any of the squashed commits had their own PRs, those PRs will become stale and need to be manually closed. Git stk currently does not automatically detect or close these PRs since the commits no longer exist individually.

Best practice: After squashing, run git stk view to review your stack state and manually close any orphaned PRs on GitHub.
